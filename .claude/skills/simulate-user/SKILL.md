## Quick Start

```
/simulate-user [코드 붙여넣기]               → 프로토타입 코드(HTML/React) 기반 시뮬레이션 (추천)
/simulate-user                              → 최근 프로토타입/PRD 감지, 페르소나 자동 선택
/simulate-user [URL]                        → 목업 링크 (Figma, Lovable, v0, HTML 등) 대상 시뮬레이션
/simulate-user [prototype file path]        → 로컬 프로토타입 파일 대상 시뮬레이션
/simulate-user [feature description]        → 텍스트 설명 기반 시뮬레이션
/simulate-user --personas "신규 유저,파워 유저" → 커스텀 페르소나 지정
/simulate-user --scenario "첫 방문"          → 특정 시나리오 집중
```

**가장 추천하는 방법:** Claude Artifacts, Lovable, v0 등에서 만든 프로토타입 코드를 그대로 복붙. 코드에 유저 플로우, 상태 전환, 조건 분기가 다 들어있어서 가장 정확한 시뮬레이션이 가능하다.

**What you get:** ICP 페르소나별 UX 반응 시뮬레이션 리포트. 혼란 지점, 이탈 위험, 개선 제안 포함.

**Time:** 5-10분

**When to use:** Phase 0 Discovery/Shaping에서 프로토타입 제작 후, 실제 고객 시연 전. 빠른 1차 검증으로 프로토타입 이터레이션 속도를 높인다.

## Context Routing Logic (Internal - for Claude)

**Automatic Context Checks:**

| Source | Files/Folders | What to Extract |
|--------|---------------|-----------------|
| ICP | `context-library/strategy/icp-*.md` | 페르소나 정의, JTBD, pain points, 행동 패턴 |
| Active Prototypes | `outputs/prototypes/*.md` | 프로토타입 플로우, 화면 구성 |
| Active PRDs | `outputs/prds/*.md`, `context-library/prds/*.md` | 요구사항, 유저 플로우, 엣지 케이스 |
| User Research | `context-library/research/*.md` | 실제 유저 인터뷰 인용, 행동 패턴, 불만 |
| CS Feedback | `context-library/research/cs-*.md` | 실제 고객 문의 패턴, 자주 막히는 지점 |
| Journey Maps | `outputs/journey-maps/*.md` | 기존 유저 여정, 감정 곡선 |

**Context Priority:**
1. ICP 정의 FIRST (누구의 반응을 시뮬레이션하는가)
2. 프로토타입/PRD SECOND (뭘 보여주는가)
3. 실제 유저 리서치 THIRD (시뮬레이션의 근거)
4. CS 피드백 FOURTH (현실 보정)

**Cross-Skill Links:**
- 프로토타입 없으면 → `/prototype` 또는 `/napkin-sketch` 먼저 제안
- 시뮬레이션 결과로 수정 후 → `/prototype-feedback`으로 구조화된 리뷰
- ICP가 없거나 오래됐으면 → `/ideal-customer-profile` 제안
- 시뮬레이션 후 실제 인터뷰 설계 → `/interview-guide`

## Workflow

### Step 1: 입력물 확인

프로토타입, PRD, URL, 또는 기능 설명 중 하나가 필요하다.

- **코드 복붙이면 (추천):** Claude Artifacts(HTML/React), Lovable, v0 등에서 생성한 프로토타입 코드를 직접 붙여넣기. 전체 유저 플로우, 상태 전환, 조건 분기, 에러 핸들링까지 파악 가능. 스크린샷보다 훨씬 풍부한 시뮬레이션 가능.
- **URL이면:** WebFetch 또는 `/browse`로 화면을 캡처하고 플로우를 파악. Figma, Lovable, v0, 배포된 HTML 등. 단, 인증이 필요한 URL(claude.ai 공유 링크 등)은 접근 불가할 수 있음.
- **파일 경로면:** 해당 파일을 읽어서 프로토타입 내용 파악
- **텍스트 설명이면:** 핵심 플로우를 정리해서 확인 후 진행
- **입력이 없으면:** 최근 `outputs/prototypes/`, `outputs/prds/` 스캔해서 후보 제시

**입력 방식별 시뮬레이션 품질:**

| 입력 | 플로우 파악 | 상태 전환 | 엣지 케이스 | 추천도 |
|------|-----------|----------|-----------|--------|
| 코드 복붙 | 전체 | 조건 분기까지 | 에러/빈 상태 포함 | 최고 |
| URL (배포된 사이트) | 탐색 가능 | 클릭해서 확인 | 일부 | 좋음 |
| 스크린샷 | 한 화면 | 불가 | 불가 | 보조 |
| 텍스트 설명 | PM 설명 의존 | 명시된 것만 | 명시된 것만 | 최소 |

### Step 2: 페르소나 선택

ICP 파일에서 페르소나를 로드한다. 기본 페르소나 세트:

| 페르소나 | 설명 | 시뮬레이션 관점 |
|----------|------|----------------|
| **Creator (ICP 핵심)** | CPG 리서처, 3~10년차, 리서치 설계~보고서 직접 수행 | "이걸로 내 일이 빨라지나?" 효율성, 전문가 기능 |
| **Viewer** | PM/마케터/임원, Creator가 만든 인사이트 소비자 | "한눈에 이해되나?" 직관성, 의사결정 지원 |
| **신규 유저** | Dataspace 처음 접하는 사람, 온보딩 경험 | "뭘 해야 하는지 알겠나?" 학습 곡선, 첫인상 |
| **Enterprise 관리자** | IT/보안 담당, 조직 관리 관점 | "우리 조직에 안전하게 도입 가능한가?" 권한, 보안, 관리 |

PM이 `--personas`로 커스텀 페르소나를 지정하면 해당 페르소나로 대체/추가.

### Step 3: 시나리오 설정

각 페르소나별로 시뮬레이션할 시나리오를 설정한다.

**기본 시나리오:**
- **첫 접근:** 이 화면/플로우를 처음 보는 순간
- **핵심 태스크:** 메인 JTBD를 수행하는 과정
- **엣지 케이스:** 빈 상태, 대량 데이터, 권한 경계, 에러 상황
- **반복 사용:** 두 번째, 세 번째 사용 시 경험

PM이 `--scenario`로 특정 시나리오를 지정하면 해당 시나리오에 집중.

### Step 4: 시뮬레이션 실행

**각 페르소나별로 서브에이전트를 병렬 실행한다.**

각 에이전트는 해당 페르소나의 관점에서:

1. **플로우 워크스루:** 화면/단계를 순서대로 밟으며 반응
2. **내면 독백:** 각 단계에서 유저가 실제로 생각할 것 (1인칭)
3. **감정 곡선:** 각 단계의 감정 상태 (기대/중립/혼란/좌절/만족)
4. **행동 예측:** 실제로 취할 행동 (클릭, 이탈, 도움 요청, 무시 등)
5. **혼란 지점 플래그:** "여기서 막힌다" 포인트 명시

**시뮬레이션 품질 규칙:**
- 실제 유저 리서치 데이터가 있으면 반드시 근거로 사용. 상상이 아니라 데이터 기반.
- CS 피드백에 비슷한 패턴이 있으면 인용. "실제로 고객 X가 이런 문의를 했음"
- ICP의 JTBD와 pain point를 기준으로 반응 생성. 일반적인 UX 원칙이 아니라 이 유저의 맥락.
- 긍정적 반응도 포함. "여기서 뭔가 되는 느낌"도 중요한 시그널.

## 시뮬레이션 vs 실제 유저 테스트

이 스킬은 실제 유저 테스트를 **대체하지 않는다.** 보완한다.

| | 에이전트 시뮬레이션 | 실제 유저 테스트 |
|---|---|---|
| **속도** | 5-10분 | 며칠~주 |
| **비용** | 0 | 고객 미팅 조율 비용 |
| **강점** | 빠른 이터레이션, 엣지 케이스 탐색, 다수 페르소나 동시 | 예상 못한 행동 발견, 감정의 진짜 강도 |
| **약점** | ICP 데이터 기반 추론이라 blind spot 존재 | 느리고, 샘플 수 제한 |
| **용도** | 1차 필터링 (명백한 문제 제거) | 2차 확정 (방향 검증) |

**원칙:** 시뮬레이션으로 "이건 확실히 문제다"를 빠르게 잡고, "이게 정말 되나?"는 실제 유저에게 확인.

## Tips

- **리서치 데이터가 많을수록 시뮬레이션 품질이 올라간다.** `context-library/research/`에 인터뷰 데이터가 있으면 시뮬레이션이 현실에 가까워짐.
- **시뮬레이션 결과를 프로토타입 수정 후 재실행하면** 개선 효과를 바로 확인 가능.
- **"이 시뮬레이션이 틀릴 수 있는 지점"을 항상 확인할 것.** 리포트 마지막에 blind spot 섹션이 있음.
- **실제 유저 인터뷰 후 시뮬레이션 결과와 비교하면** 시뮬레이션 정확도가 올라감 (calibration).
