# /ask-nova — Nova 관점 시뮬레이션

> PM이 작성한 산출물을 Nova의 관점에서 시뮬레이션 리뷰한다.
> "Nova라면 여기서 뭐라고 할까?"를 미리 점검하는 셀프 캘리브레이션 도구.

---

## When to Use

- PRD, 분석 문서, 슬랙 메시지, 의사결정 문서를 Nova에게 보내기 전
- "이 수준이면 괜찮은가?" 확신이 안 설 때
- 방향성이 맞는지 사전 점검하고 싶을 때

---

## Activation

사용자가 `/ask-nova`를 입력하거나 "Nova 관점에서 봐줘", "Nova한테 보내기 전에 체크해줘" 같은 요청을 하면 이 스킬을 실행한다.

---

## Step 1: 입력 확인

1. 리뷰할 산출물을 확인한다 (파일 경로, 붙여넣기, 또는 최근 outputs/ 파일)
2. 산출물 타입을 판별한다: PRD / 분석 문서 / 슬랙 메시지 / 의사결정 문서

---

## Step 2: Nova 관점 로드

`context-library/nova-playbook.md`를 읽고, 산출물 타입에 맞는 리뷰 기준을 적용한다.
`context-library/nova-prd-review-criteria.md`도 함께 참조한다.

**중요:** playbook의 기준을 그대로 나열하지 않는다. 기준을 이 산출물에 적용해서 구체적인 피드백을 생성한다.

---

## Step 3: 경험자 캘리브레이션

이 스킬의 대상은 3년 이상 경험의 PM이다. 초보자 코칭이 아니라 전략적 캘리브레이션을 한다.

### Nova가 경험자에게 묻는 질문의 수준

경험자에게 Nova는 기본기를 확인하지 않는다. 대신 이런 질문을 던진다:

| 영역 | Nova가 경험자에게 기대하는 것 |
|------|-------------------------------|
| 문제 정의 | "이게 지금 풀어야 할 가장 중요한 문제가 맞나?" — 표면 문제를 넘어 근본 문제로 리프레이밍했는가 |
| 전략 연결 | 한 줄 언급이 아니라, Growth loop이나 Lock-in 구조에 이 기능이 어떻게 기여하는지 설명 |
| 스코프 | 다 넣은 게 아니라, Phase로 나눠서 검증 단위를 분리했는가. Kill/Go 지점이 있는가 |
| 메트릭 | "TBD"가 아니라, 가설에 직결되는 정량 기준. 이 숫자가 안 나오면 kill한다는 선언 |
| 에비던스 | 있으면 구체적으로 인용. 없으면 "⚠️ 직접적인 evidence 0건"으로 정직하게 명시 |
| 경계 정의 | 이 PRD가 다루는 것과 다루지 않는 것의 경계가 명확한가 |

### 흔한 갭 패턴 (경험자도 빠지는 함정)

- **구조/기능 중심 서술** → 가치/가설 중심으로 전환 필요
- **전략 연결이 선언적** → 구체적 루프나 메커니즘으로 연결 필요
- **스코프가 한 덩어리** → 검증 가능한 Phase 단위로 분리 필요
- **문제를 그대로 수용** → "이게 진짜 문제인가?" 한 번 더 질문 필요
- **인접 영역과의 관계 미정의** → 이 PRD의 경계를 명시적으로 선언 필요

---

## Step 4: 시뮬레이션 리뷰 실행

### PRD 리뷰

nova-playbook.md의 PRD 리뷰 기준(1~4순위)을 이 산출물에 적용한다.
각 기준에 대해:
- ✅ 충분함 → 한 줄 확인
- ⚠️ 보강 필요 → 구체적으로 뭘 어떻게 고치면 되는지
- 🔴 방향 재검토 → Nova와 먼저 논의할 사안

### 분석 문서 리뷰

nova-playbook.md의 "산출물을 볼 때 보는 것" 기준을 적용한다.

### 슬랙 메시지 / 상향 보고

핵심이 첫 줄에 있는지, 액션이 명확한지, 결정 vs 공유 구분이 되는지.

### 의사결정 문서

옵션 비교가 같은 기준인지, 추천안과 이유가 있는지, 열린 질문이 분리되어 있는지.

---

## Step 5: 판정 및 액션 제안

### 종합 판정

| 판정 | 의미 |
|------|------|
| ✅ **바로 보내도 됨** | Nova가 볼 때 추가 질문 없이 진행될 수준 |
| ⚠️ **수정 후 보내기** | 아래 체크리스트 반영하면 됨. 방향은 맞음 |
| 🔴 **Nova와 먼저 논의** | 방향 자체를 확인해야 함. 수정이 아니라 대화가 필요 |

### 수정 체크리스트

⚠️ 판정일 때, Nova 피드백 패턴(인정 → 원칙 → 액션)을 따라서:
1. 잘된 부분을 먼저 짚는다
2. 각 수정 사항에 원칙을 붙인다 (왜 이게 중요한지)
3. 구체적 체크박스 액션을 준다
4. "지금 안 해도 되는 것"을 명시한다

---

## Output Format

```markdown
# Nova 관점 시뮬레이션 — [산출물 제목]

## 종합 판정: [✅ / ⚠️ / 🔴]

[한 줄 요약]

## 리뷰 상세

### [기준 1]
[판정 + 피드백]

### [기준 2]
...

## 수정 체크리스트
- [ ] [액션 1]
- [ ] [액션 2]

## 지금 안 해도 되는 것
- [항목]
```

---

## Notes

- 이 스킬은 Nova를 **대체**하지 않는다. 셀프 점검 도구다.
- 🔴 판정이 나오면 실제로 Nova와 대화하는 게 맞다.
- `context-library/`에 전략 문서, ICP, 확정 PRD가 많을수록 시뮬레이션 정확도가 올라간다.
