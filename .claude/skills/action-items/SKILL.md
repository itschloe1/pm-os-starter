## Quick Start

```
/action-items                              → 가장 최근 미팅록에서 추출
/action-items [미팅록 파일 경로]             → 특정 미팅록에서 추출
/action-items [회의 내용 붙여넣기]           → 바로 처리
```

**Input:** 미팅록(이미 정리된 것), 회의 스크립트, 또는 방금 `/meeting-notes`로 처리한 결과물
**Output:** 토픽별 체크박스 액션아이템 → `outputs/meeting-notes/[date]-[topic]-액션아이템.md`

---

## How This Skill Works

이 스킬은 회의 내용에서 "누가 뭘 해야 하는지"를 뽑아내되, 단순 나열이 아니라 **PM이 팀에 바로 전달할 수 있는 형태**로 만든다. 핵심은 세 가지:

1. **토픽별 그룹핑** - 회의에서 논의된 주제를 A, B, C... 토픽으로 묶는다
2. **체크박스 + 서브불릿** - 각 액션은 체크박스, 구체적 실행 내용은 서브불릿
3. **PM 판단 코멘트** - "고민 안 해도 되는 것", "로드맵 연결", "미결정 사항" 등 맥락 추가

팀원에게 이 파일을 보내면 각자 자기 항목만 보고 바로 실행할 수 있어야 한다. 표나 단락이 아니라 **스캔 가능한 체크리스트**가 목표다.

---

## Step 1: Input 확인

미팅록 소스를 확인한다:

1. **파일 경로가 주어진 경우** → 해당 파일 읽기
2. **내용이 직접 붙여넣기된 경우** → 바로 처리
3. **아무것도 없는 경우** → `outputs/meeting-notes/`에서 가장 최근 파일 찾기. 없으면 물어보기

```
가장 최근 미팅록을 찾았습니다: outputs/meeting-notes/[filename]
이 회의에서 액션아이템을 뽑을까요?
```

---

## Step 2: Context 확인

액션아이템의 질을 높이려면 관련 맥락이 필요하다. 미팅록에서 언급된 키워드로 다음을 빠르게 확인한다:

| 소스 | 확인할 것 | 왜 |
|------|----------|-----|
| `context-library/strategy/*.md` | 관련 로드맵, 스프린트 계획 | 기존 일정과 교차 확인 |
| `context-library/prds/*.md` | 관련 PRD | 기존 의사결정과 충돌 여부 |
| `context-library/decisions/*.md` | 이전 의사결정 | 중복/충돌 방지 |
| `outputs/meeting-notes/*.md` | 이전 미팅 액션아이템 | 반복되는 항목 식별 |

모든 소스를 다 읽지 말 것. 미팅록에서 언급된 프로젝트/제품명으로 관련 파일만 빠르게 스캔한다.

---

## Step 3: 액션아이템 추출 + 포맷팅

### 출력 포맷

아래 구조를 정확히 따른다:

```markdown
# [회의 제목] - 액션아이템

> 회의일: [YYYY-MM-DD] ([요일])
> 참석: [참석자 목록]
> 원본: [미팅록 파일 경로]

---

## A. [토픽 제목]

[1-2문장 맥락 설명. 이 토픽이 왜 중요한지, 회의에서 어떤 결론이 나왔는지.]

- [ ] **[액션아이템 제목]** (@담당자)
  - [구체적 실행 내용]
  - [제약조건, 옵션, 고려사항]
- [ ] **[액션아이템 제목]**
  - [상세]
  - [상세]

[선택적 PM 판단 코멘트]

---

## B. [다음 토픽]
...
```

### 포맷 규칙

**토픽 헤더:**
- `## A.`, `## B.`, `## C.` 순서로 알파벳 라벨
- 제목은 짧고 명확하게 (예: "기억력 DB 구축", "LLM 모델 + 응답 전략")
- 헤더 바로 아래에 1-2문장 맥락. 이 토픽이 왜 나왔는지, 핵심 결론이 뭔지

**체크박스 항목:**
- `- [ ] **액션아이템 제목**` 형태. 제목은 볼드
- 담당자가 명확하면 `(@이름)` 추가
- 서브불릿은 2칸 들여쓰기 후 `-`
- 서브불릿에는: 구체적 스펙, 선택지, 제약조건, 관련 수치 등

**선택지가 있는 경우:**
회의에서 여러 안이 논의됐으면 안 별로 정리한다:
```markdown
- [ ] **[의사결정 주제]**
  - 안 1: [설명]
    - [세부사항]
  - 안 2: [설명]
    - [세부사항]
  - [ ] 각 안의 개발 워크사이즈 파악 (개발팀 확인)
```

**PM 판단 코멘트 (토픽 끝에 선택적으로 추가):**

이 부분이 단순 회의록 정리와 차별화되는 핵심이다. 팀원이 불필요하게 고민하는 걸 줄여준다.

| 코멘트 유형 | 용도 | 예시 |
|------------|------|------|
| 고민 안 해도 되는 것 | 범위 밖임을 명시 | "DB를 완벽하게 만들 필요 없음. 문항 맵 먼저." |
| 로드맵 연결 | 기존 일정/계획과 교차 | "3/17 kill/go 결정의 선행조건." |
| 미결정 사항 | 후속 논의 필요한 것 | "담당자 미정. 랩업 미팅에서 배정." |
| 선행조건 | 이것 먼저 해야 다음 가능 | "커버리지 측정 완료 후에만 기억력 가중치 실험 가능." |
| 비용/일정 참고 | 투자 규모 감각 | "비용 추적 가능한 구조로 (HY)." |

코멘트는 볼드 없이, 본문과 동일한 톤으로 짧게 쓴다. 없어도 되는 토픽에는 넣지 않는다.

---

## Step 4: 토픽 구성 원칙

### 그룹핑 기준

회의 내용을 토픽으로 묶을 때:

1. **의사결정 단위로 묶는다** - "누가 뭘 결정해야 하는가" 기준
2. **실행 단위로 묶는다** - 같은 사람/팀이 실행할 것끼리
3. **시간순이 아니다** - 회의에서 나온 순서가 아니라, 중요도/긴급도 순

### 토픽 순서

1. 선행조건이 되는 과제 먼저 (이걸 안 하면 뒤가 막히는 것)
2. 공수가 큰 과제
3. 의사결정이 필요한 과제
4. 후속/관리 과제 (미팅 잡기, 일정 확인 등)는 마지막

### "후속 미팅 + 로드맵 정합" 토픽

거의 모든 회의에서 마지막 토픽으로 포함된다:
- 미결정 사항 정리
- 후속 미팅 필요 여부
- 기존 로드맵/일정과의 정합성 확인

---

## Step 5: 저장 + 안내

파일을 저장하고 요약을 보여준다:

```
`outputs/meeting-notes/[date]-[topic]-액션아이템.md`에 저장했습니다.

**토픽 구성:**
- A. [토픽] - 체크박스 N개
- B. [토픽] - 체크박스 N개
- ...

**담당자별:**
- @이름1: N개 항목
- @이름2: N개 항목
- 미배정: N개 항목 (랩업 미팅에서 배정 필요)
```

---

## 품질 체크리스트

출력 전에 확인:

- [ ] 모든 체크박스에 실행 가능한 동사가 있는가 ("확인", "구축", "설계", "테스트")
- [ ] 담당자가 명확한 항목은 (@이름)이 있는가
- [ ] 담당자가 불명확한 항목은 "담당자 미정"으로 표시했는가
- [ ] 서브불릿이 구체적인가 (무엇을, 어떻게, 어떤 기준으로)
- [ ] 토픽 맥락 설명이 "왜 이걸 해야 하는지"를 담고 있는가
- [ ] 관련 로드맵/PRD/이전 결정과 교차 확인했는가
- [ ] "고민 안 해도 되는 것"으로 범위를 좁혀줬는가 (해당되는 토픽에서)

## Cross-Skill Integration

**Input으로 받을 수 있는 것:**
- `/meeting-notes` 출력물 (가장 일반적)
- `/meeting-cleanup` 출력물 (여러 미팅 한꺼번에 처리한 경우)
- 직접 붙여넣은 미팅 스크립트/노트

**Output이 연결되는 곳:**
- `/create-tickets` → 체크박스 항목을 Jira/Linear 티켓으로
- `/slack-message` → 담당자별 할 일을 슬랙으로 전달
- `/daily-plan` → 오늘 할 일에 미팅 액션아이템 반영
