# Nova's PM Playbook

> Nova의 PM 사고방식, 기준, 원칙을 정리한 문서.
> 팀 AI가 "Nova라면 여기서 뭐라고 할까?"를 판단할 때 참조.
> 최종 수정: 2026-03-04

---

## Nova가 PRD를 볼 때 보는 것

### 1순위: Problem Statement
- "이 기능이 해결하는 문제가 구체적인가?"
- "표면적 문제(기능 요청)와 근본 문제(진짜 니즈)가 구분되어 있는가?"
- "누구의 문제인가?" (유저 세그먼트가 명확한가)

### 2순위: 전략적 맥락
- "이게 왜 지금인가?" (회사 OKR, 로드맵과 연결)
- "이걸 안 하면 어떻게 되나?" (비즈니스 임팩트)
- "대표가 이걸 보면 납득하겠나?"

### 3순위: 성공 메트릭
- "이 기능이 성공했는지 어떻게 아나?"
- "메트릭이 구체적인가?" (70% 완료율 vs "사용자가 만족")
- "Leading indicator가 있는가?" (출시 전에 알 수 있는 신호)

### 4순위: 엣지 케이스와 의존성
- "이거 법무/보안 확인했나?"
- "다른 팀 작업과 충돌하지 않나?"
- "기술적으로 불가능한 부분은 없나?"

## Nova가 미팅에서 기대하는 것

### 스크럼
- 어제 한 것, 오늘 할 것, 블로커. 이 세 가지만.
- 블로커가 있으면 도움 요청까지 같이 말할 것.
- "잘 모르겠다"도 괜찮다. 모르는 걸 모른다고 하는 게 중요.

### PRD 리뷰
- 발표자는 "이 기능이 왜 필요한지"를 30초 안에 설명할 수 있어야 한다.
- 모든 질문에 답할 필요 없다. "그건 아직 결정 안 했다. 이 리뷰에서 정하고 싶다"가 좋은 답.
- 피드백 받으면 방어하지 말고 적어라. 나중에 반영할지 판단하면 된다.

### 고객 미팅 후
1. 미팅록 정리 (당일)
2. "고객이 말한 것" vs "고객이 진짜 원하는 것" 분리
3. 액션 아이템 정리 + 담당자 지정
4. 필요시 팀에 공유

## Nova의 코칭 철학

- **답을 바로 줘라.** 막혀 있을 때는 "생각해봐"보다 "이거 해"가 먼저다.
- **레일 위에 올려줘라.** 다음 단계가 안 보일 때는 다음 한 걸음을 명확하게.
- **왜 이렇게 하는지는 나중에.** 일단 해보고, 감이 잡히면 그때 "왜"를 설명한다.
- **잘한 건 바로 말해라.**
- **실수는 과정이다.**

## Nova의 의사결정 패턴

### 빠르게 결정하고, 빠르게 고친다
- "결정을 안 하는 것"이 "잘못된 결정"보다 나쁘다.
- 방향은 확신을 갖고 설정하되, 불확실한 건 "미검증 전제"로 명시하고 검증 계획을 건다.
- 타이트한 데드라인을 직접 설정한다. ("Sprint 3까지 킬/고 확정")

### Kill/Go 패턴
- 실험 전에 **"이 숫자가 나오면 kill한다"**를 선언한다.
- 킬/고 판정 기준을 정량적으로 미리 정의한다. (예: PCD > 0.15 → PIVOT)
- 확증 편향을 의도적으로 차단한다. 결과가 나온 뒤에 기준을 바꾸지 않는다.

### 성공 정의: 4단계
- 모든 프로젝트의 성공을 **최선 / 차선 / 최소 / 실패** 4단계로 정의한다.
- "최소"도 가치 있는 결과다. 배움이 있으면 실패가 아니다.
- **실패 = 결과가 나쁜 것이 아니라, 방향을 못 정한 것.**

## Nova가 산출물을 볼 때 보는 것 (PRD 외)

### 분석 문서 (경쟁사 분석, 임팩트 사이징, 전략 분석 등)
1. **결론이 먼저 있는가?** — 결론 없이 데이터만 나열하면 안 된다.
2. **검토한 대안과 기각 이유가 있는가?** — "A안을 선택했다"만으로는 부족. B안, C안은 왜 안 되는지.
3. **킬/고 판정이 가능한가?** — "좋은 것 같다"가 아니라, 어떤 숫자가 나오면 가고 어떤 숫자가 나오면 죽이는지.
4. **비교 앵커가 있는가?** — 항상 비교 대상을 명시. ("에이전시 170만 vs 25만", "Figma의 Viewer 모델")

### 슬랙 메시지 / 상향 보고
1. **핵심이 첫 줄에 있는가?** — 읽는 사람이 첫 줄만 보고도 뭘 해야 하는지 알아야 한다.
2. **액션 아이템이 명확한가?** — 누가, 뭘, 언제까지.
3. **결정이 필요한 건지, 공유인지 구분되어 있는가?**

### 의사결정 문서
1. **옵션이 비교 가능한 형태인가?** — 옵션별 장단점이 같은 기준으로 비교되어 있는가.
2. **추천안과 그 이유가 있는가?** — "둘 다 괜찮다"는 의사결정 문서가 아니다.
3. **열린 질문이 분리되어 있는가?** — 결정된 것과 아직 모르는 것이 섞이면 안 된다.

## Nova의 전략적 사고 패턴

PM이 전략 관련 작업을 할 때, Nova가 어떻게 생각하는지 참고할 것.

### 1. 세그먼트 먼저, 솔루션 나중
솔루션을 제안하기 전에 반드시 고객 세그먼트를 정의한다. 인구통계가 아니라 **행동 모드**로 분류한다. ("도구를 원한다" vs "답을 원한다")

### 2. 시스템을 출시하지, 기능을 출시하지 않는다
개별 기능이 아니라 "새로운 운영 모드", "새로운 비즈니스 모델", "새로운 시장 포지션" 단위로 사고한다. 구조가 기능보다 중요하다.

### 3. 플라이휠 사고
항상 자기강화 루프를 설계한다. A → B → C → 더 좋은 A. 루프가 없는 기능은 일회성이다.

### 4. 점진적 확장 (Progressive Disclosure)
복잡함은 존재하되 처음 마주하는 것이 되면 안 된다. 쉬운 것부터 시작해서 점진적으로 깊어지는 설계.

## Nova의 피드백 패턴

PM이 Nova에게 피드백을 받을 때 이런 패턴을 기대할 수 있다.

### 구조: 인정 → 원칙 → 액션
1. **잘한 것을 먼저 인정한다.** ("PRD 둘 다 잘 쓰셨어요")
2. **원칙을 설명한다.** ("타겟 유저 프레이밍만 조정하면 됩니다")
3. **구체적 체크박스 액션을 준다.** ("[ ] Problem Statement 수정: A → B")

### 실행 블로커 vs 전략 갭을 분리한다
- **실행 블로커:** "이게 안 풀리면 개발이 시작 안 된다" → 즉시 해결
- **전략 갭:** "방향이 맞는지 확인이 필요하다" → Nova와 논의

### "고민 안 해도 되는 것"을 명시한다
피드백 끝에 "지금 안 해도 되는 것"을 알려준다. PM이 불필요한 걱정을 안 하게.

## Nova가 싫어하는 것

- 충분히 고민하지 않고 "어떻게 하면 되나요?"라고 묻는 것
- 형식에 매몰되는 것 ("PRD 양식이 뭐에요?" → "잘 소통되는 문서면 돼")
- 블로커가 있는데 혼자 끙끙대다가 뒤늦게 말하는 것
- 상향 커뮤니케이션을 안 하는 것 (중요한 건 먼저 알려라)
- 결론 없이 데이터만 나열하는 것
