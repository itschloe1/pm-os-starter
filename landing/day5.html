피처 개발 프로세스

읽기 15분 + 실습 30분 /day5

Day 1~4에서는 매일 쓰는 루틴 스킬을 배웠어요. Day 5에서는 피처 하나가 아이디어에서 론치까지 가는 전체 흐름을 배웁니다. 각 단계에서 어떤 스킬을 쓰는지, 다음 단계로 넘어가려면 뭐가 충족돼야 하는지를 익혀보세요.

각 단계를 아래에서 하나씩 살펴봅니다

Phase 0

Discovery / Shaping

"이걸 만들 가치가 있나?" 를 검증하는 단계. 문제를 정의하고, 기회 크기를 추정하고, 프로토타입으로 빠르게 검증합니다.

스킬 순서
1
/impact-sizing copy
기회 크기 추정
2
/generate-ai-prototype copy
프로토타입 제작
3
/simulate-user copy
페르소나 시뮬레이션

이 단계가 끝나면

  • 문제가 해결할 가치가 있는지 검증됨
  • 프로토타입으로 방향 검증됨 (AI 시뮬레이션 + 실제 피드백)
  • PRD를 쓸지, 프로토타입+Constraint Doc으로 갈지 판정
Tip: /generate-ai-prototype으로 프로토타입 만들고 → /simulate-user로 ICP 페르소나 반응 시뮬레이션 → 수정 → 다시 시뮬레이션. 이 루프를 빠르게 돌린 후 실제 고객에게 시연하세요.
Phase 1

PRD Lock

"뭘 만들고, 왜 만드는가"를 정렬하는 단계. PRD를 작성하고 리뷰어 피드백을 반영해서 스코프를 확정합니다.

1
/prd-draft copy
PRD 초안 작성
2
/prd-review-panel copy
멀티 관점 리뷰

이 단계가 끝나면

  • 리뷰어 전원 피드백 반영 완료
  • 스코프 (In/Out) 확정
  • 성공 지표 + Kill criteria 정의됨
  • PM이 Lock 선언
Phase 2

설계 (SRS + Figma)

"PRD 대로 만들 수 있나?"를 확인하는 단계. TL이 SRS를 작성하고, UX가 Figma를 만듭니다. PM은 트래킹하고 Open Question을 해소합니다.

이 단계에서는 별도 스킬 없이 PM이 직접 챙깁니다:
• SRS가 PRD 스코프와 맞는지 확인
• Figma 리뷰
• 개발/QA와 요구사항 확정일 합의
• 보안 검토 요청 (해당 시)

이 단계가 끝나면

  • SRS + Figma 확정
  • 요구사항 확정일 합의 (개발/QA 서명)
  • Open Questions 해소됨
Phase 3

스프린트 (개발)

실제로 만드는 단계. PM은 티켓을 생성하고, 진행을 트래킹하고, 블로커를 해소합니다.

1
/create-tickets copy
Jira 티켓 생성

이 단계가 끝나면

  • 스프린트 목표 기능 개발 완료
  • 론치 체크리스트 초안 완성
  • Analytics 이벤트 구현 확인
Phase 4

검증 (QA + 보안)

"론치해도 되나?"를 확인하는 단계. QA 테스트, 보안 검토, 내부 Dogfood, 롤백 계획을 점검합니다.

1
/launch-checklist copy
론치 체크리스트 생성

이 단계가 끝나면

  • Critical bug 0건
  • 보안 검토 통과
  • 롤백 계획 + 기준 문서화
  • Kill criteria 대시보드 라이브
  • 론치 Go/No-Go 판정
Phase 5

론치

"고객에게 어떻게 전달하나?" CS 브리핑, 세일즈 브리프, 고객 커뮤니케이션을 준비하고 배포합니다.

1
/sales-brief copy
세일즈 브리프
2
/slack-message copy
팀/고객 안내
3
/status-update copy
스테이크홀더 업데이트

이 단계가 끝나면

  • Comms doc 완성, 마케팅 핸드오프
  • CS 브리핑 완료
  • Sales Brief 세일즈팀에 공유
  • Feature flag on, 모니터링 라이브
Phase 6

사후 (임팩트 리뷰)

"잘 됐나? 계속할까?" 론치 후 1주, 4주에 걸쳐 메트릭을 리뷰하고, 계속/중단/개선을 판정합니다.

1
/feature-results copy
론치 결과 분석
2
/weekly-review copy
주간 회고에 반영

이 단계가 끝나면

  • Primary Metric 판정 완료
  • Go/Kill/Iterate 판정 (계속 / 중단 / 개선 후 재시도)
  • Lessons learned 문서화
  • 다음 iteration 백로그 정리
